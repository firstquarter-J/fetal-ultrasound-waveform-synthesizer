# 리팩터링 분석 및 단계별 계획

## 목적

이 문서는 현재 저장소 구조 분석 결과와, 이후 단계별로 진행할 리팩터링 계획을 고정하기 위한 기준 문서입니다.

진행 현황 기준:

- 완료: 0단계 기준선 고정
- 완료: 1단계 계약과 공개 API 분리
- 완료: 2단계 `timeseries` 경로 분리
- 다음 기본 단계: 3단계 CV 경로와 모션 분석 분리

이 프로젝트의 최종 목적은 다음과 같습니다.

- 태아 초음파 영상에서 파형 구간 검출
- 검출된 파형을 기반으로 태아 심박음 생성

따라서 리팩터링의 기준도 "코드를 예쁘게 나누는 것"이 아니라 아래를 만족하는 방향이어야 합니다.

- 검출 정확도와 재현성 유지
- 검출 결과를 합성 단계가 소비하기 쉬운 구조로 정리
- 이후 합성 엔진을 붙여도 공개 계약이 흔들리지 않도록 경계 설정

## 현재 상태 요약

- 샘플셋 기준으로 파형 구간 검출은 1차 목표를 달성한 상태로 판단합니다.
- 기본 공개 경로는 `timeseries` 기반 구간 검출입니다.
- 기본 출력은 프레임 단위 라벨 집합이 아니라 "검출 구간 리스트"입니다.
- 심박음 합성 모듈은 아직 실질 구현이 없습니다.

샘플셋 기준 참고 지표:

- 평균 recall: `0.9104`
- 평균 precision: `0.9530`
- 알려진 약한 케이스:
  - `28w-126bpm.mp4`는 실제 시작보다 늦게 검출되는 경향이 있음

## 현재 구조 분석

### 1. 검출 모듈 책임 과밀

현재 핵심 로직은 `src/detection/analyzer.py` 한 파일에 과도하게 집중되어 있습니다.

한 파일 안에 아래 책임이 함께 들어 있습니다.

- 전역 파라미터 정의
- ROI 추출
- 시계열 특징 추출
- 시계열 기반 구간 검출
- 프레임 기반 CV 검출
- 모션 분석
- 구간 병합과 후처리
- 공개 API

문제점:

- 일부 로직만 수정해도 전체 파일을 함께 건드리게 됨
- 합성 단계가 붙을 때 출력 계약 안정성이 낮아짐
- 시계열 경로와 CV 경로의 경계가 흐림

### 2. 검출과 합성 사이의 계약 부재

현재 기본 반환값은 dict 기반 구간 리스트이며, 구조가 코드 내부 구현과 강하게 결합되어 있습니다.

문제점:

- 합성 모듈이 붙으면 필드 이름 변경이 바로 연쇄 영향으로 이어짐
- 평가 스크립트와 합성 입력 형식이 같은 계층에서 섞일 가능성이 큼
- "검출 결과가 무엇인지"를 문서가 아니라 구현이 정의하고 있음

### 3. 스크립트와 실험 도구가 같은 레벨에 혼재

`scripts/` 아래에 현재 아래 성격의 파일이 함께 있습니다.

- 사용자 실행용 엔트리포인트
- 배치 분석 스크립트
- 수동 라벨 평가 도구
- 비교 리포트 생성 도구
- 파라미터 튜닝 도구

문제점:

- 실제 운영용 스크립트와 실험용 스크립트의 경계가 약함
- README와 실제 진입점이 다시 어긋날 가능성이 있음

### 4. 라벨/산출물의 위치 정리는 진행 중

현재 수동 라벨 문서는 `annotations/`로 분리되었고, 주요 평가/비교 결과도 `results/`로 모으는 방향으로 정리되고 있습니다.

문제점:

- 입력 라벨과 생성 결과의 구분은 좋아졌지만, 남은 레거시 산출물이 있는지 계속 관리가 필요함
- 평가 루틴이 늘어날수록 루트가 복잡해질 가능성이 큼

## 리팩터링 원칙

- 한 단계씩 진행합니다.
- 각 단계는 되돌릴 수 있어야 합니다.
- 각 단계가 끝날 때 기존 검출 동작이 깨지지 않아야 합니다.
- 공개 진입점은 가능하면 유지합니다.
- 구조를 바꾸더라도 샘플셋 평가 기준이 크게 악화되면 안 됩니다.
- 합성 단계 준비에 직접 도움이 되는 변화부터 우선합니다.

## 단계별 계획

### 0단계. 기준선 고정

목적:

- 리팩터링 전후 비교 기준을 명확히 합니다.

작업:

- 현재 공개 API와 주요 스크립트 동작을 문서 기준으로 고정
- 샘플셋 평가 결과를 기준선으로 유지
- 리팩터링 중 확인할 최소 검증 항목 정리

완료 기준:

- `analyze_video()`의 현재 반환 구조를 기준 계약으로 정리
- 샘플 실행과 `py_compile` 검증 루틴이 정리됨

### 1단계. 계약과 공개 API 분리

목적:

- 검출 내부 구현과 외부 소비 구조를 분리합니다.

작업:

- `src/contracts/waveform.py` 생성
- `WaveformSegment` 같은 검출 결과 계약 정의
- `src/detection/api.py` 생성
- 현재 `analyze_video()` 진입점을 `api.py` 중심으로 재배치
- 스크립트가 내부 구현이 아니라 공개 API만 바라보도록 정리

완료 기준:

- 스크립트가 `src.detection.api`를 사용
- 공개 반환 구조가 코드상 명시됨
- 기존 단일 영상/배치/평가 스크립트 동작 유지

이 단계에서 하지 않을 것:

- 시계열 로직 대규모 재작성
- 합성 로직 구현

### 2단계. timeseries 경로 분리

목적:

- 현재 기본 경로인 `timeseries` 검출 로직을 독립 모듈로 분리합니다.

작업:

- `src/detection/timeseries/config.py`
- `src/detection/timeseries/roi.py`
- `src/detection/timeseries/features.py`
- `src/detection/timeseries/segmenter.py`
- `analyzer.py` 내부의 timeseries 관련 함수와 상수 이동

완료 기준:

- `timeseries` 경로가 독립 모듈로 분리됨
- 기존 기본 검출 결과가 실질적으로 유지됨
- `analyzer.py`의 크기와 책임이 눈에 띄게 줄어듦

이 단계에서 하지 않을 것:

- CV 경로와 모션 분석까지 동시에 정리하지 않음

### 3단계. CV 경로와 모션 분석 분리

목적:

- 프레임 기반 탐지와 후처리를 분리해 역할 경계를 명확히 합니다.

작업:

- `src/detection/cv/frame_detector.py`
- `src/detection/cv/motion.py`
- `src/detection/cv/postprocess.py`
- CV 경로 관련 헬퍼 함수 이동

완료 기준:

- CV 경로가 `timeseries` 경로와 물리적으로 분리됨
- 모션 분석과 세그먼트 후처리 책임이 구분됨

### 4단계. 라벨과 결과물 정리 마무리

목적:

- 입력 데이터와 생성 산출물의 위치 정리를 마무리합니다.

작업:

- 평가/비교 결과 기본 출력 위치를 `results/`로 통일
- 루트에 생성되는 결과 파일 후보를 재검토
- 스크립트 기본 경로와 README 동기화

완료 기준:

- 사람이 보기에 "입력 라벨", "실행 스크립트", "생성 결과"가 폴더 기준으로 구분됨

### 5단계. 합성 모듈 골격 준비

목적:

- 검출 결과를 심박음 생성 단계로 연결할 최소 구조를 만듭니다.

작업:

- `src/contracts/heartbeat.py`
- `src/synthesis/api.py`
- `src/synthesis/mapper.py`
- `src/synthesis/generator.py`
- 검출 결과를 합성 입력으로 변환하는 최소 데이터 흐름 정의

완료 기준:

- 합성기 구현 전이라도 "검출 결과 -> 합성 계획" 경로가 구조적으로 존재함

## 우선순위 판단

가장 먼저 해야 할 단계는 1단계입니다.

이유:

- 지금 가장 부족한 것은 폴더 미관이 아니라 계약 안정성입니다.
- 합성 단계가 붙기 전에 검출 결과 구조를 명시해야 이후 변경 비용이 낮아집니다.
- 1단계가 끝나야 2단계와 3단계의 내부 분해가 안전해집니다.

## 각 단계 공통 검증 항목

- `./venv/bin/python -m py_compile ...` 통과
- `scripts/analyze_single.py` 실행 가능
- `scripts/analyze_all_save.py` 실행 가능
- `scripts/evaluate_manual.py` 실행 가능
- 샘플셋 기준 주요 검출 구간이 크게 깨지지 않음

## 메모

- 현재 구조는 "연구용 검출 코드"로는 충분히 작동합니다.
- 하지만 합성 단계로 넘어가려면 "검출 결과를 안정적으로 넘길 수 있는 구조"가 먼저 필요합니다.
- 이후 작업은 이 문서를 기준으로 한 단계씩 진행합니다.
